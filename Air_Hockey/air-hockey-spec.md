## 1. 專案總覽

| 項目 | 內容 |
|---|---|
| 遊戲名稱 | Air Hockey（空氣曲棍球） |
| 遊戲類型 | 雙人對戰型體育模擬遊戲，本階段僅開放「玩家 VS AI」 |
| 執行環境 | 純前端靜態網頁，無需伺服器、無需 build 工具 |
| 開啟方式 | 直接雙擊 `index.html` 於瀏覽器開啟即可遊玩 |
| 支援裝置 | 桌上型電腦、筆電（滑鼠／鍵盤）、手機、平板（觸控） |
| 支援語言 | 中文（繁體）、英文、日文 |
| 開發限制 | 不得產生 `README.md`，遊戲完成後才另行產出 |

### 1.1 核心設計原則

- **零安裝、零建置**：不使用 Webpack / Vite / npm build，所有 JS 以原生 `<script>` 標籤或 ES Module 方式引入，CSS 以 `<link>` 引入。
- **離線可玩**：所有素材（圖片、音效、字型）皆放在專案資料夾內，不依賴外部 CDN（避免離線或無網路時失效）。若使用網頁字型，需附上本地字型檔並以 `@font-face` 載入。
- **模組化**：CSS 與 JavaScript 皆依功能拆分為多檔案，`index.html` 僅負責引入與畫面骨架，不寫行內邏輯。
- **可讀性優先**：所有文字採用大字體、高對比配色，避免任何看不清楚文字或圖示的情況。

## 2. 技術架構與執行方式

### 2.1 技術選型

| 類別 | 技術 | 說明 |
|---|---|---|
| 標記語言 | HTML5 | 單一入口 `index.html` |
| 樣式 | CSS3（原生，模組化多檔） | 使用 CSS 變數（Custom Properties）管理主題色，支援多套配色主題切換 |
| 邏輯 | JavaScript（ES6+ 原生，模組化多檔，使用 `<script type="module">`） | 不依賴任何前端框架（不使用 React/Vue），確保零 build 即可執行 |
| 繪圖引擎 | HTML5 Canvas 2D API | 桌台、球、球拍、特效皆以 Canvas 繪製，確保跨裝置效能一致 |
| 音訊 | Web Audio API + `<audio>` 元素 | BGM 與音效皆使用 Web Audio API 控制音量與混音 |
| 儲存 | `localStorage` | 儲存語言設定、音量設定、配色主題、遊戲進度（繼續遊戲用） |
| 字型 | 本地端 Web Font（OTF/WOFF2） | 內建於 `assets/fonts/`，不依賴 Google Fonts CDN |

### 2.2 執行方式規範

- 使用者只需將整個專案資料夾放置於任意位置，**直接雙擊 `index.html`**，即可以 `file://` 協定於瀏覽器開啟並完整遊玩。
- 所有資源路徑一律使用**相對路徑**（如 `./css/main.css`、`./js/core/game.js`），禁止使用絕對路徑或需要伺服器解析的路徑。
- 若瀏覽器對 `file://` 下的 ES Module 有 CORS 限制（例如部分 Chrome 版本），需提供 fallback：
  - 主要方案：`<script type="module" src="./js/main.js"></script>`
  - 相容性備援：文件內加註「若模組載入失敗，可改用瀏覽器右鍵『開啟檔案』或使用 Firefox 開啟」，或將關鍵初始化邏輯改以傳統 `<script defer>` 依序引入，避免強制依賴 module CORS 行為。
- 不使用任何 `import ... from 'npm-package'` 的裸模組寫法（bare specifier），避免需要 bundler。

## 4. RWD（響應式網頁設計）規格

### 4.1 斷點定義

| 裝置類型 | 寬度範圍 | 對應 CSS 檔 |
|---|---|---|
| 手機（直向／橫向） | ≤ 767px | `mobile.css` |
| 平板 | 768px – 1024px | `tablet.css` |
| 桌機／筆電 | ≥ 1025px | `desktop.css` |

### 4.2 RWD 核心規範

- 遊戲畫布（Canvas）以**等比例縮放**方式適應容器，使用 `aspect-ratio` 或 JS 動態計算，確保球台比例（建議 2:3，長:寬）在任何裝置上不變形。
- 所有 UI 元件（按鈕、HUD、彈窗）使用相對單位（`vw`、`vh`、`rem`、`%`）而非固定 `px`，確保縮放一致。
- 手機直向模式：建議引導使用者橫向持機遊玩，顯示一個「請將手機橫向」的提示畫面（僅在偵測到直向且進入遊戲時顯示，主選單不受限）。
- **禁止任何 UI 元素遮擋遊戲主畫面**：
  - 比分／計時 HUD 固定於畫面**最上方安全區域**（不超過畫面高度 8%）。
  - 行動裝置虛擬控制區域（見第 9 章）固定於畫面**左右兩側或底部透明浮動層**，不覆蓋球台範圍，且採用半透明設計，觸控時才提高不透明度。
  - 使用 CSS `env(safe-area-inset-*)` 處理 iPhone 瀏海／手勢列造成的安全區域問題。
- 遊戲進行中畫面需支援視窗縮放（`resize` 事件）即時重繪 Canvas，不可造成拉伸模糊或裁切。
- 觸控裝置需停用瀏覽器預設的「雙指縮放」「長按選單」「下拉刷新」等手勢（`touch-action: none;` + `preventDefault()`），避免誤觸中斷遊戲。

### 4.3 效能友善的 RWD

- 手機裝置預設使用較低的粒子特效數量（見第 14 章擬真特效），依裝置效能自動降級，確保流暢度優先於畫面華麗度。

## 5. 視覺設計規範（字體、配色、風格）

### 5.1 字體規範

- **全站字體一律採用大字體**，基準字級規範如下：

| 元素 | 桌機字級 | 平板字級 | 手機字級 |
|---|---|---|---|
| 主標題（LOGO／頁面標題） | 64px | 48px | 36px |
| 按鈕文字 | 32px | 28px | 24px |
| 內文說明文字 | 24px | 22px | 20px |
| 比分／計時 HUD 數字 | 48px | 40px | 32px |
| 提示／次要文字 | 20px | 18px | 16px |

- 字體需**清晰明確**，中文採用 Noto Sans TC Bold、日文採用 Noto Sans JP Bold、英文數字採用 Poppins Bold，皆為粗體以確保辨識度。
- 文字一律加上陰影或描邊（`text-shadow` 或 `-webkit-text-stroke`），避免與背景色融合導致看不清楚。
- 行高（line-height）不得小於 1.4，確保多語系換行不擁擠。

### 5.2 配色規範

- 提供**多套可切換的配色主題**（於設定頁選擇），至少包含以下 4 種：

| 主題名稱 | 主色調 | 適用情境 |
|---|---|---|
| 霓虹（Neon） | 桃紅＋電光藍＋深紫背景 | 活潑動感風格 |
| 經典藍（Classic） | 寶藍＋白＋銀灰 | 傳統球台風格，最接近真實空氣曲棍球台 |
| 日落橘（Sunset） | 橘紅＋金黃＋深藍背景 | 溫暖對比風格 |
| 冰藍（Ice） | 冰藍＋白＋淺灰 | 清爽冷色風格，模擬冰面質感 |

- **對比度規範**：所有文字與背景對比度須符合 WCAG AA 標準以上（對比比例 ≥ 4.5:1），主要互動元素（按鈕、可點擊圖示）對比比例 ≥ 3:1。
- 每套主題皆需分別定義：背景色、球台色、球台邊界色、玩家球拍色、AI 球拍色、冰球色、HUD 背景色、按鈕主色／hover 色／disabled 色，並以 CSS 變數統一管理（例如 `--color-primary`、`--color-table`、`--color-mallet-player`），切換主題時只需替換對應的 `theme-*.css` 檔或動態切換 CSS class。
- 嚴禁出現「深色文字疊深色背景」「亮色圖示疊亮色背景」等低對比組合，設計階段需逐一檢查每個畫面在四套主題下的可視性。

### 5.3 視覺風格

- 整體風格走「明亮、乾淨、具科技感」路線，避免過度花俏導致視覺疲勞。
- 使用圓角設計（`border-radius`）、柔和陰影（`box-shadow`）、按鈕按下時的縮放回饋動畫（`transform: scale()`），提升現代感與觸覺回饋。

## 6. 主畫面規格

### 6.1 主畫面功能項目

主畫面**僅提供以下四項功能**，保持畫面簡潔：

1. **開始遊戲**：進入難度選擇畫面（簡單／普通／困難），選擇後開始新的一局。
2. **繼續遊戲**：僅當 `localStorage` 中存在未完成的對戰進度時才顯示為可點擊狀態；若無存檔則顯示為灰階不可點擊，並標註提示文字（依語系顯示，如「尚無進度」）。
3. **說明**：進入遊戲說明頁面（第 12 章）。
4. **設定**：進入設定頁面（第 13 章）。

### 6.2 主畫面設計限制

- **主畫面不得包含任何可調整設定的選項**（例如音量滑桿、語言切換、配色選擇皆不可直接出現在主畫面），所有調整項目一律收整於「設定」頁面內，確保主畫面乾淨。
- 主畫面允許呈現：遊戲 LOGO／標題、背景動畫（例如緩慢漂浮的冰球粒子特效或球台光暈呼吸動畫）、版本號（小字放置於角落）。
- 語言切換入口例外處理：可在主畫面右上角放置一個小型「地球圖示＋當前語言縮寫」按鈕，點擊後彈出語言選單（僅限語言切換，不算作「設定選項」，因語言影響全站文字顯示，需第一時間可調整）；若團隊希望完全遵守「不可有設定選項」，亦可將語言切換移入設定頁，僅在首次啟動時跳出語言選擇彈窗。**建議採用「首次啟動語言選擇彈窗＋設定頁內可再次修改」的方案**，主畫面上不常駐任何設定按鈕。

### 6.3 主畫面按鈕行為

- 四個功能按鈕垂直置中排列，按鈕尺寸與間距在不同裝置下等比例縮放。
- 每個按鈕搭配對應圖示（▶ 開始、⏸ 繼續、❓ 說明、⚙ 設定），圖示＋文字並列，避免僅靠圖示造成理解困難。
- 按鈕 hover／focus／active 狀態皆需有明顯視覺回饋（顏色加深、輕微放大、陰影加強）。

## 7. 遊戲核心玩法規格（VS AI）

### 7.1 基本規則

- 本階段**僅開放「玩家 VS AI」模式**，不含雙人本機對戰或連線對戰。
- 球台採俯視 2D 視角，玩家控制下方球拍，AI 控制上方球拍。
- 球台左右各有一個球門缺口，冰球進入對方球門得 1 分。
- 預設先得 **7 分**者獲勝（可於設定頁調整為 5 / 7 / 10 分制，屬於遊戲規則設定，非畫面必要選項，放置於設定頁的「遊戲規則」區塊）。
- 每次得分後，冰球重置於球台中央，倒數 3 秒（伴隨倒數音效）後重新開球。

### 7.2 物理規則

- 冰球具備摩擦力係數（可微調），使其在無碰撞時逐漸減速，避免永動不真實。
- 球拍與冰球碰撞採用**動量守恆**簡化公式計算反彈角度與速度，撞擊球拍邊緣時產生偏折效果，增加策略性。
- 冰球撞擊球台邊界時以入射角＝反射角原則彈開，並觸發撞牆音效與輕微火花特效。
- 球拍移動限制在己方半場範圍內，不可越過中線（中線可視覺化為一條發光分隔線）。

### 7.3 遊戲流程

1. 選擇難度 → 顯示 3 秒倒數 → 開球。
2. 雙方球拍即時互動，物理引擎持續運算碰撞。
3. 任一方得分 → 顯示得分特效與音效 → 短暫慶祝動畫（0.8–1.2 秒）→ 重置開球。
4. 任一方達到目標分數 → 顯示勝負結果畫面（含「再玩一次」「返回主畫面」按鈕）。
5. 遊戲過程中可隨時點擊暫停鈕，彈出暫停選單（繼續／回主選單／重新開始），暫停時自動存檔進度供「繼續遊戲」使用。

## 8. AI 難度設計（簡單／普通／困難）

| 難度 | 反應速度 | 預測能力 | 移動速度上限 | 失誤率 | 設計說明 |
|---|---|---|---|---|---|
| 簡單（Easy） | 慢（約 400ms 延遲反應） | 僅追蹤冰球目前位置，不預測軌跡 | 較低 | 高（約 25% 機率延遲反應或反應錯誤方向） | 適合新手熟悉操作與物理手感 |
| 普通（Normal） | 中等（約 200ms 延遲） | 具備基礎軌跡預測（預測冰球接下來 0.3 秒位置） | 中等 | 中（約 10% 失誤率） | 提供有來有往的對戰節奏 |
| 困難（Hard） | 快（約 60ms 延遲，接近即時） | 完整彈道預測（含撞牆反彈路徑計算），並具備主動進攻與封堵策略 | 高（接近人類極限反應速度） | 低（約 2% 失誤率） | 具挑戰性，考驗玩家走位與假動作 |

- 三種難度皆須避免 AI 出現「無視物理限制的瞬移」或「必勝不公平演算法」，需維持公平競技感，讓玩家有機會取勝。
- AI 邏輯獨立拆分為 `ai-easy.js`／`ai-normal.js`／`ai-hard.js` 三個檔案，共用同一介面（例如 `decideNextMove(gameState)`），方便未來擴充或調整平衡性。
- 難度選擇畫面需以清楚圖示區分（例如簡單＝綠色笑臉、普通＝黃色持平表情、困難＝紅色鬥志表情），搭配大字體標籤文字。

## 9. 操作方式（滑鼠／觸控／行動裝置按鍵）

### 9.1 桌機操作

- 滑鼠移動直接對應玩家球拍位置（球拍跟隨滑鼠座標，限制在己方半場與球台邊界內）。
- 可選鍵盤方向鍵／WASD 作為輔助操作（於設定頁可開關）。

### 9.2 觸控操作

- 手指觸控並拖曳即可控制球拍，支援多點觸控偵測但僅取第一個觸控點為有效輸入。
- 觸控靈敏度可於設定頁調整（低／中／高）。

### 9.3 行動裝置虛擬按鍵規範

- 由於本遊戲操作方式以「拖曳球拍」為主，行動裝置**不需要額外的方向搖桿**，但仍需提供以下浮動 UI 按鍵，且**不得遮擋球台與冰球活動範圍**：
  - 左上角：暫停鈕（半透明圓形圖示，觸控時才提高不透明度）。
  - 右上角：音效靜音快捷鈕（半透明圖示）。
- 上述按鍵一律置於**球台範圍之外的畫面邊角安全區**，並使用 `touch-controls.css` 統一管理尺寸與透明度，預設透明度 60%，觸控互動時提升至 100%，放開後 1 秒漸淡回 60%，兼顧可見性與不遮擋畫面的需求。
- 橫向手機模式下，因可視高度有限，虛擬按鍵尺寸需縮小並貼齊螢幕邊緣角落，避免佔用中央球台可視區域。

## 10. 音效與音樂規格

### 10.1 BGM 規範

- **風格**：全站 BGM 一律採用「鋼琴演奏、輕快明亮」風格，節奏適中偏快，營造愉悅競技氛圍，避免使用過於沉重或電子感過強的曲風。
- 分為兩軌：
  - `menu-theme.mp3`：主選單／說明／設定頁面播放，循環播放（loop）。
  - `gameplay-theme.mp3`：進入對戰畫面後切換播放，循環播放，音樂節奏可比選單稍快以提升緊張刺激感。
- BGM 切換畫面時需**淡入淡出**（約 0.5–1 秒 crossfade），避免生硬切斷。

### 10.2 音效規範

- **風格**：所有音效統一採用「高音、輕脆」音色（例如清脆鈴聲、玻璃敲擊聲、鋼片琴音色），避免低沉厚重的音效，維持整體輕快活潑調性。
- 音效清單：

| 音效 | 觸發時機 | 音色描述 |
|---|---|---|
| `hit-wall.mp3` | 冰球撞擊球台邊界 | 高音清脆「叮」聲 |
| `hit-mallet.mp3` | 球拍擊中冰球 | 高音清脆「咚」聲，比撞牆音略短促 |
| `goal-score.mp3` | 得分瞬間 | 高音上揚音階（如鋼片琴琶音），帶勝利感 |
| `button-click.mp3` | 任何按鈕點擊 | 短促高音「滴」聲 |
| `countdown-beep.mp3` | 開球倒數 3-2-1 | 高音電子「嗶」聲，第 3 聲（開球瞬間）音調更高 |

### 10.3 音量規格（重點需求）

- **遊戲中的 BGM 音量須放大為原先基準音量的 10 倍**，透過 Web Audio API 的 `GainNode` 進行增益控制實作，範例邏輯如下：

```js
// audio-manager.js 概念示意
const BASE_BGM_VOLUME = 0.05;      // 基準音量（避免預設破音的安全值）
const GAMEPLAY_BGM_MULTIPLIER = 10; // 遊戲中音量倍率需求：放大 10 倍

function setGameplayBgmVolume(gainNode) {
  const targetVolume = Math.min(BASE_BGM_VOLUME * GAMEPLAY_BGM_MULTIPLIER, 1.0);
  // 因 GainNode 實際輸出音量上限為 1.0（100%），
  // 「放大 10 倍」以基準值極低（例如 0.05）為前提換算，
  // 避免直接 10 倍導致音量爆表或破音，換算後仍需限制在可播放安全範圍。
  gainNode.gain.setTargetAtTime(targetVolume, audioContext.currentTime, 0.05);
}
```

- 實作時需特別注意：由於瀏覽器音量上限為 100%（增益值 1.0），若基準音量設定過高，直接乘以 10 倍會造成破音或瀏覽器警告，因此**基準 BGM 音量須刻意設定為極低值（例如 5%）**，讓「放大 10 倍」後的實際音量落在合理可播放範圍（約 50% 上下），此設計需在 `audio-manager.js` 中以常數清楚註解說明，方便後續調整。
- 設定頁仍保留 BGM／音效獨立音量滑桿（0–100%），此滑桿為使用者手動再次調整的「最終輸出音量」，會與上述 10 倍增益邏輯疊加運算後輸出，滑桿本身不需要讓使用者理解內部倍率換算，只需呈現直覺的百分比操作介面。
- 音效（SFX）音量不受此 10 倍規則影響，維持一般正常音量範圍，避免高音音效過於刺耳。

## 11. 多國語系（i18n）規格

### 11.1 支援語言

| 語言代碼 | 語言 |
|---|---|
| `zh` | 繁體中文 |
| `en` | 英文 |
| `ja` | 日文 |

### 11.2 實作方式

- 採用簡易 key-value 語系檔架構，每個語言各自獨立為一個 JS 檔（`lang-zh.js`／`lang-en.js`／`lang-ja.js`），格式範例：

```js
// lang-zh.js
export default {
  menu: {
    start: "開始遊戲",
    continue: "繼續遊戲",
    howToPlay: "說明",
    settings: "設定"
  },
  difficulty: {
    easy: "簡單",
    normal: "普通",
    hard: "困難"
  },
  hud: {
    player: "玩家",
    ai: "電腦",
    score: "比分"
  }
  // ...其餘所有畫面文字
};
```

- `i18n-manager.js` 負責讀取目前語系並將所有帶有 `data-i18n-key` 屬性的 DOM 元素文字即時替換，切換語言時不需重新整理頁面。
- 語言選擇需覆蓋**所有畫面文字**，包含：主選單、說明頁、設定頁、遊戲內 HUD、暫停選單、勝負結果畫面、錯誤提示訊息。
- 語言設定儲存於 `localStorage`，下次開啟自動套用上次選擇的語言；若無記錄，預設依瀏覽器 `navigator.language` 自動偵測，無法辨識則預設為英文。
- 首次啟動遊戲時彈出語言選擇畫面（3 個大按鈕：中文／English／日本語，各自以該語言呈現，避免使用者看不懂而卡關）。

3. **遊戲規則**：得分規則、球台中線規則、勝利分數說明，搭配編號步驟圖示（①②③）。

- 所有圖示需與第 5 章配色規範一致，並確保深色／淺色主題切換下皆清晰可見。
- 頁面右上角或底部提供明顯的「返回主畫面」按鈕，固定顯示不隨捲動消失（`position: sticky` 或 `fixed`）。
- 說明頁面內容需完整支援三國語系切換。

## 14. 遊戲畫面擬真與視覺豐富度

### 14.1 擬真元素

- 球台材質採用貼近真實空氣曲棍球台的視覺（可選經典藍主題作為擬真基準），包含：
  - 球台中線發光虛線、四角落弧形導角。
  - 球台邊框具備立體光影（`box-shadow` 內外陰影模擬桌台厚度）。
  - 球門區域以凹陷陰影表現深度感。
- 冰球與球拍加入輕微陰影投射（模擬懸浮於氣墊上的效果），移動時可加入淡出拖尾效果，增加速度感與真實感。
- 撞擊瞬間加入微小螢幕震動（`transform: translate` 微幅抖動 100ms 內）與火花／光暈粒子特效，特效數量依設定頁「特效品質」調整。

### 14.2 豐富有趣的視覺元素

- 得分時球門處觸發慶祝動畫（彩帶粒子、閃光擴散圈）。
- 連續得分（Combo）可觸發額外文字特效（例如「Nice!」「連續得分！」以動畫彈跳文字呈現），需支援多語系文字。
- 主選單背景加入緩慢漂浮的裝飾性冰球光點動畫，避免主畫面過於死板。
- 勝負結算畫面依勝負結果呈現不同氛圍（獲勝：金色彩帶＋歡快音效；落敗：鼓勵性文字＋柔和音效，避免過度負面觀感）。

### 14.3 配色可視性檢查（呼應第 5.2 節）

- 特效顏色（火花、彩帶、光暈）需與當前配色主題背景做對比檢查，避免出現「白色特效疊白色背景」等看不見的狀況，建議特效顏色統一使用**高飽和度亮色＋深色描邊**組合，確保在四套主題下皆清晰可辨。

## 15. 效能與流暢度規範

- 遊戲主迴圈統一使用 `requestAnimationFrame`，目標為**穩定 60 FPS**（行動裝置低效能情況下至少維持 30 FPS 以上不掉幀卡頓）。
- 物理運算（`physics.js`）與畫面渲染分離，避免渲染邏輯阻塞碰撞運算。
- Canvas 繪製採用**離屏渲染（Offscreen Canvas／分層繪製）**策略：靜態背景（球台、線條）只需繪製一次並快取，動態物件（球拍、冰球、特效）才逐幀重繪，降低每幀運算量。
- 所有圖片素材需最佳化壓縮（PNG 使用適當壓縮、避免過大解析度），音訊採用 MP3／OGG 壓縮格式，控制整體專案資源大小，確保低階裝置也能快速載入。
- 觸控與滑鼠事件監聽需節流（throttle）處理，避免高頻事件造成運算堆積導致掉幀。
- 特效系統（粒子、光暈）需設定**最大同時存在數量上限**，並於低效能模式（設定頁「特效品質：低」）大幅減少粒子數與陰影運算，確保行動裝置操作依然流暢不卡頓。
- 視窗縮放（`resize`）事件需做防抖（debounce）處理，避免拖曳視窗過程中頻繁觸發重新計算造成卡頓。

## 16. 資料儲存與存讀檔（繼續遊戲）

- 使用 `localStorage` 儲存以下資料：
  - `settings`：音量、配色主題、語言、觸控靈敏度、特效品質等設定值。
  - `gameProgress`：目前對戰中的比分、難度、剩餘時間（若有）、暫停狀態，供「繼續遊戲」功能使用。
- 對戰進度僅在**暫停或非正常關閉頁面時**（`beforeunload` 事件）自動存檔；正常結束一局（分出勝負）後需清除進度存檔，主畫面「繼續遊戲」按鈕自動變回不可點擊狀態。
- 存檔格式建議使用 JSON 字串化儲存，範例：

```json
{
  "difficulty": "normal",
  "playerScore": 3,
  "aiScore": 2,
  "targetScore": 7,
  "timestamp": 1735689600000
}
```

- 讀取存檔失敗（格式錯誤、版本不符）時需靜默處理並視為無存檔，不可讓遊戲白屏或報錯崩潰。

```
   └─ 設定 ──▶ [設定頁面] ──▶ (返回) ──▶ [主畫面]

   [遊戲進行中]
   ├─ 得分 ──▶ [得分特效] ──▶ (未達勝分) 回到遊戲進行中
   │                         └─(達到勝分) ──▶ [結算畫面] ──▶ 再玩一次 / 返回主畫面
   └─ 暫停鈕 ──▶ [暫停選單] ──▶ 繼續 / 重新開始 / 回主畫面（自動存檔）
```

- 畫面切換一律由 `screen-manager.js` 統一管理，採用單一 DOM 容器＋顯示/隱藏對應 `<section>` 的方式（不使用路由跳轉頁面，確保純前端單頁運作零延遲）。
- 畫面切換時需有淡入淡出轉場動畫（約 200–300ms），提升操作質感，並避免動畫期間可重複點擊觸發多次切換（切換期間鎖定輸入）。
