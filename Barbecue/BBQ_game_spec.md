## 1. 專案概述

| 項目 | 內容 |
|---|---|
| 遊戲名稱 | Barbecue 烤肉樂園（暫名，可於設定中顯示為多語系標題） |
| 遊戲類型 | 休閒模擬 / 烹飪操作類（Cooking Simulation / Casual Arcade） |
| 核心玩法 | 玩家操作烤肉架，翻烤肉品、控制火候、避免烤焦，完成訂單得分 |
| 目標裝置 | 桌機瀏覽器（Chrome / Edge / Safari / Firefox）＋ 行動裝置瀏覽器（iOS Safari / Android Chrome） |
| 執行方式 | **純前端、零依賴、零建置**，雙擊 `index.html` 即可遊玩 |
| 語系支援 | 中文（繁體）／英文／日文，可即時切換 |
| 風格定位 | 可愛療癒風（Kawaii / Cute）＋ 略帶擬真質感的燒烤場景 |

### 1.1 核心設計原則
- **零安裝門檻**：不使用任何 npm、webpack、vite 等建置工具，不需 `npm install`、不需啟動 server。
- **易讀性優先**：文字一律加大加粗，避免小字造成閱讀困難。
- **行動裝置友善**：操作按鍵絕不遮擋遊戲主要視覺區域。
- **聽覺愉悅**：BGM 與音效皆走輕快、清脆、療癒路線，並依需求大幅提高遊戲內 BGM 音量。

## 2. 技術架構與執行方式

### 2.1 技術限制
- **不可使用**：Node.js build 工具（Webpack / Vite / Parcel）、npm scripts 執行、任何需要終端機指令才能啟動的方案、任何後端 server（Node/Express/PHP 等）。
- **可使用**：原生 HTML5 + CSS3 + Vanilla JavaScript（ES6+）。
- **可使用（僅限離線可用）**：Canvas API、Web Audio API、LocalStorage、CSS Animation / Transition、SVG。
- **外部資源**：所有圖片、音檔、字體皆需**內含於專案資料夾內**（相對路徑引入），不可依賴需要網路連線的 CDN，確保離線雙擊也能執行。

### 2.2 啟動流程
```
使用者操作：
1. 下載/ 取得專案資料夾
2. 直接雙擊 index.html（或用瀏覽器開啟 index.html）
3. 遊戲立即載入並可遊玩，無需任何額外步驟
```

### 2.3 index.html 引入原則
`index.html` 僅負責「結構＋引入」，不得內嵌大量 inline CSS/JS（僅允許極少量的 critical loading 樣式）。所有樣式與邏輯皆從對應分類資料夾以 `<link>` 與 `<script>` 引入，並依「先共用、後模組」順序載入：

## 4. RWD 響應式設計規範

### 4.1 斷點定義

| 裝置類型 | 寬度範圍 | 版面策略 |
|---|---|---|
| 手機（直式） | ≤ 480px | 單欄、遊戲畫面置頂滿版、操作鍵置底浮動列 |
| 手機（橫式）/ 小平板 | 481px–768px | 遊戲畫面置中，操作鍵左右分離放置畫面兩側下緣 |
| 平板 | 769px–1024px | 遊戲畫面加大，UI 面板可並排顯示 |
| 桌機 | ≥ 1025px | 遊戲畫面置中固定比例，兩側可顯示裝飾性留白插畫 |

### 4.2 核心規則
1. **遊戲主畫面（Canvas）永遠置中，且比例鎖定（建議 4:3 或 16:10）**，以 `max-width` / `max-height` 搭配 `aspect-ratio` 控制，任何裝置都不會變形。
2. **行動裝置操作按鍵**一律使用「懸浮半透明底層 + 圖示按鈕」，固定於畫面**下方或左右兩側邊緣的安全區域**，且按鈕區域**不可與遊戲有效視覺範圍重疊**（透過保留 Canvas 下方 / 側邊 12%~15% 的留白來實現，而非蓋在畫面上）。
3. 使用 `safe-area-inset`（`env(safe-area-inset-bottom)` 等）避免 iPhone 瀏海/手勢列遮擋按鈕。
4. 觸控熱區（touch target）最小 **48x48px**，按鈕間距最少 8px，避免手機誤觸。
5. 直向／橫向切換時即時重新排版（監聽 `orientationchange` / `resize`），無需重新整理頁面。
6. 文字、圖示大小以 `clamp()` 或 `vw` 為單位彈性縮放，確保小螢幕也維持「大字體、看得清楚」原則。

### 4.3 版面示意（文字描述）
```
┌─────────────────────────────┐
│   頂部資訊列（分數/時間/語言） │  ← 固定高度，不隨畫面滾動
├─────────────────────────────┤
│                             │
│      遊戲主畫面 Canvas       │  ← 置中滿版，比例鎖定
│      （烤肉架＋食材）        │
│                             │
├─────────────────────────────┤
│  [翻面] [加大火] [調味] [出餐] │ ← 行動裝置浮動按鍵列，半透明底
└─────────────────────────────┘   絕不疊加在食材/烤架有效範圍上
```

## 5. 字體與配色規範

### 5.1 字體規範
- 全站字級一律採「大字體」策略：
  - 內文最小 **18px（行動裝置）／ 20px（桌機）**
  - 標題級距：H1 40px／H2 32px／H3 26px
  - 按鈕文字最小 20px，且加粗（`font-weight: 700`）
- 字型：使用圓潤可愛風字型（如圓體 / 可愛手寫圓角字型），中日英三語皆需各自對應可讀性佳字型並內嵌於 `assets/fonts/`，避免因缺字型跳回系統預設造成排版跑版。
- 所有文字需與背景保持足夠對比度（WCAG AA 以上），避免「看不清楚文字」情形。

### 5.2 配色方案（可於設定中切換，共 4 組）

| 方案名稱 | 主色 | 輔色 | 強調色 | 情境 |
|---|---|---|---|---|
| 經典木炭橘（預設） | #FF7A45 | #FFF3E0 | #E85D2C | 溫暖木炭火烤感 |
| 粉彩馬卡龍 | #FFB6C1 | #FFF0F5 | #7FC8A9 | 可愛甜美風 |
| 夜間野營 | #2C3E50 | #F1C40F | #E67E22 | 星空露營夜烤 |
| 海島清涼 | #00B4D8 | #CAF0F8 | #FF9F1C | 夏日海邊烤肉 |

- 每組配色皆確保：按鈕文字/圖示與底色對比清楚、狀態色（成功=綠、警告=黃、失敗/烤焦=紅）在四組配色中都維持一致辨識度。
- 提供「可愛風主題」作為視覺總風格（不是配色本身），大量搭配：愛心、星星、雲朵、笑臉食材、圓角卡片、貼紙式邊框、彈跳動畫圖示，套用在任一配色方案之上。

## 6. 主畫面（首頁）規格

### 6.1 版面原則
- **主畫面必須簡潔乾淨**，只保留 4 顆核心按鈕，**不可放置任何設定選項（如音量調整、語言切換等）直接顯示在主畫面上**，一律收整進「設定」頁面。
- 背景為可愛風插畫（庭院烤肉場景 + 緩慢飄動的煙霧/雲朵動畫），營造豐富但不干擾按鈕的氛圍。

### 6.2 主畫面按鈕（4 顆，垂直排列置中）

| 按鈕 | 功能說明 |
|---|---|
| 🔥 開始遊戲 | 建立全新進度，直接進入食材/難度選擇後進入遊戲 |
| ⏯️ 繼續遊戲 | 讀取 LocalStorage 中最後一次存檔進度；若無存檔則此按鈕顯示為淺灰不可點擊狀態並提示「尚無存檔」 |
| 📖 說明 | 開啟遊戲玩法說明頁（Modal 或獨立頁面） |
| ⚙️ 設定 | 開啟設定頁，內含音量、畫質、語言、配色主題等選項 |

- 按鈕皆為大尺寸圓角卡片式設計，搭配對應可愛圖示（火焰、播放符號、書本、齒輪），並有輕微 hover/tap 彈跳動畫回饋。
- 主畫面右上角僅保留一個「語言切換」小圖示（地球圖示 🌐），因語言為進入遊戲前就需決定的必要體驗設定，其餘設定仍全數收於「設定」頁。

## 7. 遊戲主玩法規格

### 7.1 玩法核心
1. 畫面中央為烤肉架（Canvas 繪製格狀鐵網），玩家可放置食材（肉片、香腸、玉米、蝦子、蔬菜等）。
2. 每項食材有「生 → 半熟 → 全熟（最佳）→ 微焦 → 烤焦（失敗）」共 5 個視覺與數值階段，隨時間與火力進度推進。
3. 玩家需在「全熟」階段內點擊食材進行「翻面」或「起鍋」，起鍋過早或過晚都會影響得分。
4. 訂單系統：畫面上方顯示可愛顧客插畫與其點餐氣泡（例如：「我要 2 份全熟牛肉！」），依語系顯示對應文字。
5. 完成訂單獲得金幣與星星評分，星星影響關卡解鎖與獎勵造型（如烤肉架花色、廚師服裝）。
6. 火力可調整（大火／中火／小火），影響食材熟成速度，玩家需策略性分配注意力於多個食材格位。

### 7.2 操作方式對應

| 平台 | 操作方式 |
|---|---|
| 桌機 | 滑鼠點擊食材＝翻面／夾取；鍵盤方向鍵可切換選取格位（可選） |
| 行動裝置 | 觸控點擊食材＝翻面／夾取；底部浮動按鍵可調整火力、召喚新食材、出餐 |

## 8. 音效與 BGM 規格

### 8.1 BGM 規範
- 採用**多首「鋼琴輕快風格」BGM**輪播，存放於 `assets/audio/bgm/`，例如：
  - `bgm_piano_morning.mp3`（晨間輕快）
  - `bgm_piano_picnic.mp3`（野餐愉快）
  - `bgm_piano_festival.mp3`（慶典歡快）
  - `bgm_piano_lullaby_upbeat.mp3`（輕搖滾明亮版）
- 主畫面、遊戲畫面、設定/說明頁可各自指定不同 BGM 或延續同一輪播清單，並支援無縫循環（loop）與淡入淡出（fade in/out）切換，避免音樂中斷感。

### 8.2 音效規範
- 音效一律採**高音、輕脆、清亮**類型（如：木琴聲、鈴鐺聲、氣泡聲、清脆點擊聲），存放於 `assets/audio/sfx/`：
  - `sfx_click.mp3`（按鈕點擊）
  - `sfx_flip.mp3`（翻面）
  - `sfx_success_star.mp3`（訂單完成、星星閃爍）
  - `sfx_perfect.mp3`（完美起鍋，音調更高更清脆）
  - `sfx_burnt_warning.mp3`（即將烤焦提醒，仍需輕快不刺耳）

### 8.3 音量控制與「遊戲內 BGM x10 音量」規則
- `audioManager.js` 需實作獨立的 **BGM 音量倍率邏輯**：
  - 使用者於設定頁調整的 BGM 音量基準值（0.0～1.0）僅套用於**主畫面／說明／設定等非遊戲進行中畫面**。
  - **當玩家正式進入「遊戲進行中」畫面時，BGM 播放音量需以基準音量 × 10 倍運算後輸出**（例如：Web Audio API 之 GainNode.gain.value = baseVolume × 10，並視瀏覽器限制做上限保護 clamp，避免破音或超出 1.0 的硬體限制時以等比例壓縮方式維持相對倍率關係，同時提供使用者於設定頁可另外快速一鍵調整「遊戲內音量」防止音量過大造成困擾）。
  - 音效（SFX）音量不受此 x10 規則影響，維持獨立音量軸線，確保音效仍清脆不過度轟炸。
- 所有音訊皆需支援使用者瀏覽器「使用者互動後才允許播放音訊」的限制（於首次點擊「開始遊戲」或任一按鈕時，透過使用者手勢觸發 AudioContext resume()）。

## 9. 多國語系（i18n）規格

### 9.1 支援語言
- 🇹🇼 中文（繁體）
- 🇺🇸 英文
- 🇯🇵 日文

### 9.2 架構設計
- `js/i18n/i18n.js`：核心切換邏輯，提供 `setLanguage(langCode)`、`t(key)` 兩個主要函式。
- `lang-zh.js` / `lang-en.js` / `lang-ja.js`：各自匯出 key-value 字串物件，例如：

```js
// lang-zh.js
window.LANG_ZH = {
  menu_start: "開始遊戲",
  menu_continue: "繼續遊戲",
  menu_help: "說明",
  menu_settings: "設定",
  order_request: "我要 {count} 份{food}！",
  status_perfect: "完美！",
  status_burnt: "烤焦了…",
};
```

- 所有畫面文字（按鈕、標題、訂單對話、提示訊息、設定選項名稱）皆須透過 `data-i18n="key"` 屬性標記，並由 `i18n.js` 於切換語言時統一掃描替換，不允許任何寫死（hardcode）的單一語言文字殘留於 UI 上。
- 語言選擇會寫入 `LocalStorage`，下次開啟遊戲自動套用使用者最後選擇的語言。
- 日文與英文文案需為**該語言使用者會自然閱讀的道地翻譯**，非逐字直翻（例如日文訂單語氣需符合顧客可愛口吻，如「〜をお願いします！」）。

| 分類 | 選項 | 元件形式 |

## 12. 視覺美術風格規格

### 12.1 整體風格
- 主風格：**可愛療癒風插畫**（圓潤造型、大眼睛擬人化食材與角色、粉嫩描邊）為主體介面風格。
- 場景細節可帶一定「擬真質感」：例如烤肉架的金屬光澤反光、炭火的橘紅漸層光暈、肉品油脂光澤與煙霧飄散粒子效果，讓畫面「可愛但不廉價」，兼具豐富細節與真實感。

### 12.2 圖示規範
- 所有互動圖示（按鈕、狀態提示、食材圖示）皆採手繪可愛風格，統一線條粗細與圓角弧度，禁止混用不同風格圖示（例如避免扁平極簡圖示與可愛插畫圖示混搭造成違和）。
- 常用可愛裝飾元素：星星、愛心、笑臉太陽、雲朵、煙霧棉花糖狀效果、閃亮亮片特效（完成訂單/升級時）。

### 12.3 配色與可見性檢查
- 嚴禁出現「文字與背景同色系導致看不清楚」「重要按鈕顏色過淺融入背景」等狀況；所有互動元素皆須通過對比度檢查後才定案。
- 每個配色主題上線前，需個別檢視：按鈕文字可讀性、食材熟度色階可辨識性（尤其是「全熟」與「微焦」需有明顯區隔，避免色弱使用者混淆，建議搭配圖示形狀輔助辨識，不單靠顏色判斷）。

## 13. 效能與流暢度規格

- 遊戲主迴圈使用 `requestAnimationFrame` 驅動，目標穩定 **60 FPS**（行動裝置至少維持 30 FPS 以上不掉幀卡頓）。
- Canvas 繪製採用「僅重繪有變動區域」或「分層 Canvas（背景層/食材層/UI層）」策略，避免整版全部重繪造成效能浪費。
- 圖片資源需適當壓縮並提供合理解析度（行動裝置不載入過大原圖），音檔採壓縮格式（mp3/ogg），確保載入快速、記憶體佔用低。
- 所有動畫（按鈕彈跳、煙霧飄動、星星閃爍）優先使用 CSS Transform/Opacity 或 Canvas 繪製，避免觸發大量 Reflow/Repaint。
- 行動裝置觸控事件需做防抖（debounce）與多點觸控衝突處理，避免快速連續操作造成邏輯錯亂或畫面延遲。

## 14. 存檔與繼續遊戲機制

- 使用 `LocalStorage` 儲存：目前關卡進度、金幣、星星數、已解鎖收藏、玩家設定（語言/音量/配色/操作設定）。
- 存檔時機：每完成一筆訂單、每次關卡結束、每次修改設定時自動存檔（不需玩家手動存檔）。
- 主畫面「繼續遊戲」按鈕會偵測是否存在有效存檔資料：
  - 有存檔 → 按鈕可點擊，點擊後直接恢復至上次關卡進度與場景狀態。
  - 無存檔 → 按鈕呈現淺灰不可點擊狀態，避免玩家誤觸產生錯誤。
