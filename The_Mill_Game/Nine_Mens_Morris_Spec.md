## 1. 專案總覽

### 1.1 專案目標

製作一款**純前端、零後端、零安裝**的九子棋（Nine Men's Morris / Mill Game）網頁遊戲。使用者只需要直接用瀏覽器開啟 `index.html` 即可遊玩，不需要任何打包流程（不需要 npm install、不需要 webpack/vite build、不需要啟動任何 local server）。遊戲須支援電腦瀏覽器與手機瀏覽器，畫面需完整 RWD 並維持高可玩性、高可讀性與良好的視覺美感。

| 名詞 | 說明 |
|------|------|
| Mill（磨坊 / 三連線） | 同色三顆棋子連成一條直線（棋盤上預先定義的合法連線），達成時可以移除對方一顆棋子 |

## 2. 技術架構與限制

### 2.1 技術選型

- **HTML5**：單一 `index.html` 作為唯一進入點。
- **CSS3**：純手寫 CSS，依功能拆分為多檔案，於 `index.html` 用 `<link>` 依序引入。不使用 Sass/Less 等需要編譯的語言，避免需要 build。
- **JavaScript（ES6+，原生 Vanilla JS）**：不使用任何前端框架（React/Vue 等），不使用任何需要打包工具的模組系統。所有 JS 檔案以 `<script>` 標籤依序引入，並透過全域命名空間（例如 `window.NMM`）做模組間溝通，避免使用 `import/export` ES Module 語法造成 `file://` 協定下 CORS 問題。
- **音訊**：全部使用瀏覽器原生 `Web Audio API` 即時合成（Oscillator + GainNode + Envelope），不需要任何 `.mp3`/`.wav` 外部素材檔案，確保零依賴、開啟即播放。
- **儲存**：使用 `localStorage` 儲存進度、設定。
- **圖示**：使用內嵌 `SVG` 或 Unicode Emoji／純 CSS 圖形繪製，不依賴外部圖示庫 CDN（避免使用者離線或無網路時無法載入，但允許在有網路時可選用 CDN 字型作為加分項，非必需）。

### 2.2 「免 Build / 免 Server」的硬性限制

因為規格要求使用者直接打開 `index.html`（即 `file://` 協定）就要能玩，因此實作時必須注意：

1. **不可使用 ES Module `<script type="module">` 搭配相對路徑 import**，部分瀏覽器在 `file://` 下會因 CORS 政策擋下 module 載入。改用傳統 `<script src="..."></script>` 並依賴順序載入。
2. **不可使用 `fetch()` 讀取本地 JSON/語言檔**，因為 `file://` 下 `fetch` 對本機檔案會被部分瀏覽器（尤其 Chrome）以 CORS 擋掉。語言包應直接寫成 JS 物件（`lang-zh.js`、`lang-en.js`、`lang-ja.js`），透過 `<script>` 標籤載入後存進全域變數，不透過 `fetch`/`XHR` 讀取。
3. **不可使用 Web Worker 搭配外部檔案路徑**（若使用 Worker 做 AI 運算需用 Blob URL 方式內嵌，或乾脆不使用 Worker，直接用 `setTimeout` 分段運算避免卡住主執行緒）。
4. **音訊與圖像皆不依賴外部檔案**，避免相對路徑在不同作業系統（Windows/Mac/手機）下解析差異。
5. **不使用 Service Worker**（PWA 安裝非本次需求，且 Service Worker 在 `file://` 下無法註冊）。

### 2.3 瀏覽器相容性

- 支援 Chrome、Edge、Safari、Firefox 最新兩個大版本。
- 支援 iOS Safari、Android Chrome 行動瀏覽器。
- Web Audio API 在 iOS Safari 需要「使用者第一次互動（點擊）後才能啟動 AudioContext」，因此第一個畫面（主選單）出現的「開始遊戲」等任何按鈕點擊事件，需同時觸發 `AudioContext.resume()`。

### 3.1 `index.html` 的引入規則

`index.html` 本身**只負責三件事**：(1) HTML 結構骨架（含四個 view 容器：`#view-main-menu`、`#view-game`、`#view-howto`、`#view-settings`），(2) 依順序 `<link>` 所有 CSS，(3) 依「依賴順序」`<script>` 所有 JS。不允許在 `index.html` 內寫任何 inline `<script>` 商業邏輯（僅允許最末一行呼叫 `App.init()` 作為啟動點）。

CSS 引入順序（重要，後面的可覆蓋前面的變數）：
`reset → variables → typography → app-shell → responsive → safe-area → components/* → views/* → themes/theme-classic(預設主題，其餘主題以 class 切換覆蓋變數)`。

JS 引入順序（重要，依賴關係）：
`constants → game-state → rules-engine → move-validator → ai-evaluator → ai-easy/normal/hard → ai-dispatcher → audio-engine → sfx-synth → bgm-synth → lang-zh/en/ja → i18n-manager → icons → board-renderer → hud-controller → modal-controller → theme-manager → settings-controller → howto-controller → router → save-manager → main`。

## 4. 遊戲規則設計（九子棋完整規則）

### 4.1 棋盤

- 棋盤由 **3 個同心正方形**組成（外、中、內），每個正方形 4 邊中點與相鄰正方形以直線連接，共形成 **24 個交叉點（point）**。
- 合法連線（可移動路徑）共 **24 條邊**。
- 合法「三連線（Mill）」共 **16 條**：每個正方形的 4 邊（4×3=12 條）+ 連接三個正方形中點的 4 條放射線 = 16 條。

棋盤點位編號與三連線清單詳列於附錄（第 18 章）。

### 4.2 棋子

- 雙方各持 **9 顆棋子**（玩家：例如白/亮色；AI：例如黑/暗色，顏色依所選主題變化）。

### 4.3 遊戲三階段

1. **擺子階段（Placing Phase）**
   - 雙方輪流，每次選擇一個空的點位放置 1 顆棋子。
   - 共進行 18 次放置（雙方各 9 次）後進入移動階段。
   - 若放置後形成 Mill，立即觸發吃子（見 4.5）。

2. **移動階段（Moving Phase）**
   - 雙方輪流，每次選擇己方一顆棋子，移動到與其**直接相鄰且空著**的點位（沿合法連線移動，不可跳點，不可斜走未連線的方向）。
   - 若移動後形成 Mill，立即觸發吃子。

3. **飛行階段（Flying Phase）**
   - 若某方棋子數量恰好剩下 **3 顆**，該方在輪到自己時，可以將任一棋子移動到棋盤上**任意空位**（不受相鄰限制），俗稱「飛子」。
   - 飛行規則僅適用於「剩 3 顆的那一方」，另一方若仍 >3 顆則仍依一般移動規則。

### 4.4 Mill（三連線）判定

- 每次落子/移動之後，立即檢查該枚棋子所在的所有「合法三連線」是否三點同色。
- 若同一次移動同時形成 **兩條 Mill（雙磨坊）**，視為一次達成，仍只能吃 1 顆對方棋子（除非規則變體選擇「雙磨坊吃 2 顆」，本規格採用**國際標準規則：一次只吃 1 顆**，不啟用雙磨坊吃多顆的變體，避免新手混淆，並在說明頁中註明採用標準規則）。
- **重新打開 Mill 再關閉（搖擺磨坊 / Swinging Mill）**：允許棋子移出 Mill 再移回去重新形成同一條 Mill，每次重新形成都可再次吃子（此為標準規則允許行為，不特別限制，但於「困難」AI 中會列入評估考量，避免被人類玩家以此戰術輕易擊敗）。

### 4.5 吃子（Removal）規則

達成 Mill 的一方，須立即從對方棋子中移除 1 顆，規則如下（依國際標準優先順序）：

1. **優先移除「未處於任何 Mill 中」的對方棋子。**
2. 若對方所有棋子都處於 Mill 中（無「非 Mill」棋子可選），則可以自由選擇移除任意一顆對方棋子（包含 Mill 中的棋子）。
3. 玩家進行吃子選擇時，UI 上不可選取的棋子（受保護中）需以視覺方式（半透明 + 鎖頭圖示 + 提示文字）標示，避免玩家點擊到不合法目標。

### 4.6 勝負判定

遊戲在以下情況立即結束：

| 情況 | 結果 |
|------|------|
| 任一方棋子數量被吃到只剩 **2 顆以下** | 對方獲勝 |
| 輪到某方時，該方**沒有任何合法走法**（所有棋子都被卡住、無路可走，且非飛行階段） | 對方獲勝 |
| （可選規則，預設關閉）連續 N 回合（建議 50 回合）無吃子發生 | 判定和棋（Draw），於設定頁可開關此規則，預設為**關閉**以符合最常見的標準規則認知，但說明頁需註明此為國際賽事常見的「逼和規則」 |

### 4.7 回合流程（State Machine）

```
[擺子階段]
  玩家/AI 輪流放子 → 檢查 Mill → (若 Mill) 執行吃子 → 切換回合
  → 直到 18 子全部下完 →
[移動階段]
  玩家/AI 輪流選子→選目標 → 驗證合法 → 移動 → 檢查 Mill → (若 Mill) 吃子 → 檢查勝負 → 切換回合
  → 若任一方剩 3 子 → 該方進入飛行子模式（僅該方）
  → 直到觸發勝負條件 →
[結算]
  顯示結果彈窗（勝/負）→ 提供「再玩一次」「返回主畫面」
```

- 若裝置為**橫向（landscape）且高度過矮**，則自動切換為「左右分欄」佈局（棋盤置中，玩家/AI資訊卡分置左右兩側並縮小字級），避免棋盤被壓縮過小。

### 7.3 棋盤視覺規格

- 使用 **SVG** 繪製棋盤線條與 24 個交叉點（理由：SVG 向量縮放在任何解析度都銳利清晰，且方便加上互動事件與動畫）。
- 棋盤線條顏色、交叉點顏色依「主題」變化（見 10.2）。
- 每個交叉點為可點擊的圓形熱區，半徑足夠大（建議直接點擊熱區 ≥ 44×44 px，符合行動裝置觸控標準），即使視覺上的點很小，點擊熱區仍維持容易點擊的大小。
- 棋子使用帶光澤漸層的圓形（CSS `radial-gradient` 模擬球體光影，達成「擬真」需求），玩家棋子與 AI 棋子用不同顏色 + 不同邊框強調色區分，色盲友善（額外加上棋子內部小圖示，如玩家●、AI▲ 做形狀區分，避免單純靠顏色辨識）。
- 棋子落下動畫：從上方輕微下落 + 縮放回彈（`cubic-bezier` 彈性曲線），落下時同步觸發音效。
- 形成 Mill 時：三連線上的棋子與連線**閃爍金色光暈**並有連線高亮動畫（線條描邊發光、寬度短暫加粗再恢復）。
- 可吃子目標：對方棋子上顯示「半透明紅色叉狀疊加 + 輕微脈動」表示可點擊吃掉；不可吃（受保護）的棋子顯示「半透明灰階 + 鎖頭小圖示」。
- 目前可移動的棋子（選取狀態）：套用「呼吸光暈」外框動畫；其可移動的目標空格顯示「淡綠色高亮 + 點狀導引」。

### 7.4 操作流程提示

- 畫面下方／HUD 區常駐一行「目前狀態提示文字」，使用**大字級、語意明確的句子**（非僅圖示），例如：「請選擇要移動的棋子」「請點擊空格放置棋子」「你形成了三連線！請選擇要吃掉的棋子」「電腦思考中…」。

## 10. 視覺設計系統（字體 / 配色 / RWD）

### 10.1 字體規格（對應需求 3：全站大字體、明確清晰）

採用 CSS 變數定義字級階層，全站統一引用，禁止任意 hardcode px 數值：

```css
:root {
  --font-family-base: "Noto Sans TC", "Noto Sans JP", "Noto Sans", -apple-system,
                       "Segoe UI", "Microsoft JhengHei", sans-serif;
  /* 多語系字體堆疊，確保中/英/日文字皆能正確顯示且筆畫清晰 */

  --font-size-display:  clamp(2.2rem, 6vw, 3.6rem);  /* 主標題、Logo */
  --font-size-h1:       clamp(1.6rem, 4.5vw, 2.4rem); /* 頁面大標題 */
  --font-size-h2:       clamp(1.3rem, 3.5vw, 1.8rem); /* 區塊標題 */
  --font-size-body-lg:  clamp(1.1rem, 2.8vw, 1.35rem); /* 主要內文（基準變大） */
  --font-size-body:     clamp(1rem, 2.4vw, 1.2rem);    /* 一般說明文字 */
  --font-size-button:   clamp(1.1rem, 2.6vw, 1.3rem);  /* 按鈕文字，需明確好點 */
  --font-size-caption:  clamp(0.85rem, 2vw, 1rem);     /* 最小輔助文字，仍不過小 */

  --font-weight-regular: 400;
  --font-weight-bold: 700;
  --line-height-base: 1.6;
}
```

- 全站**最小字級不低於 0.85rem（約 13.6px）**，按鈕、狀態提示等關鍵互動文字維持在 1.1rem 以上，確保「明確」可讀。
- 標題類文字使用 `font-weight: 700` 加粗，並搭配主題強調色，加強視覺層級對比。
- 使用 `clamp()` 讓字體隨視窗寬度（vw）流體縮放，同時設定最小值/最大值，避免在極窄手機或超寬螢幕上字體過小或過大。

### 10.2 配色系統（對應需求 3、11：配色多樣、不可看不見）

採用 CSS 變數 + 主題 class 切換機制，每個主題定義完整的色彩變數組，確保切換主題時**全站文字、按鈕、棋盤、背景**全部同步更新，不會出現「背景換了但文字看不見」的問題。

每個主題必須定義以下變數（範例為「經典木紋」主題）：

```css
.theme-classic {
  --color-bg-primary: #f5e6c8;       /* 主背景 */
  --color-bg-secondary: #e8d4a8;     /* 卡片背景 */
  --color-bg-elevated: #fffaf0;      /* 彈窗等浮層背景 */
  --color-text-primary: #3a2a1a;     /* 主文字（與背景對比足夠） */
  --color-text-secondary: #6b5640;   /* 次要文字 */
  --color-text-inverse: #fffaf0;     /* 深色按鈕上的文字 */
  --color-accent: #b8682f;           /* 強調色（按鈕、高亮） */
  --color-accent-hover: #9c5424;
  --color-success: #4a8c4a;          /* 成功/勝利提示 */
  --color-danger: #c0392b;           /* 警示/吃子提示 */
  --color-board-line: #5a4226;       /* 棋盤線條 */
  --color-board-bg: #d9bf8c;         /* 棋盤底色 */
  --color-piece-player: #fffaf0;     /* 玩家棋子主色 */
  --color-piece-player-shade: #cfc4ad;
  --color-piece-ai: #2b2118;         /* AI 棋子主色 */
  --color-piece-ai-shade: #000000;
  --color-mill-glow: #ffd700;        /* Mill 達成時的金色光暈 */
}
```

其餘四套主題（海洋藍綠 `theme-ocean`、夕陽暖橘 `theme-sunset`、森林綠 `theme-forest`、夜間深色 `theme-night`）皆依**相同變數命名**重新定義數值，差異只在色彩本身，確保元件 CSS 不需要因主題而改動，只需要套用對應的 `theme-*` class 於 `<body>` 上即可整站變色。

**對比度規則（避免「看不見」問題）**：

- 所有「文字色 vs 背景色」組合，皆須通過 WCAG AA 標準（對比度 ≥ 4.5:1）。
- `theme-night`（夜間主題）特別注意：背景使用深藍灰（非純黑，避免 OLED 螢幕殘影與過硬對比造成眼睛疲勞），文字使用米白色而非純白，強調色使用提高飽和度的亮色（如琉璃藍、亮黃）確保深色背景下依然清晰可辨。
- 棋子顏色與棋盤底色、與對方棋子顏色，三者互相之間對比度都需檢查，避免「白棋子在白背景上看不清楚」或「玩家棋子與AI棋子顏色太相近」的狀況。每套主題上線前需用對比度檢查工具（如 WebAIM Contrast Checker 概念）人工複檢。

### 10.3 RWD 斷點規則（對應需求 2、8：行動裝置流暢、不擋畫面）

```css
/* responsive.css 斷點規劃 */

/* 超小手機 (< 360px) */
@media (max-width: 359px) { /* 字級取 clamp 下限、HUD 高度縮減為 44px */ }

/* 一般手機直向 (360px ~ 599px) */
@media (min-width: 360px) and (max-width: 599px) { /* 預設行動版排版（見7.2） */ }

/* 平板 / 手機橫向 (600px ~ 1023px) */
@media (min-width: 600px) and (max-width: 1023px) { /* 棋盤略縮小、資訊卡並排顯示 */ }

/* 桌面 (≥ 1024px) */
@media (min-width: 1024px) { /* 完整桌面版排版（見7.1），最大寬度容器 1200px 置中 */ }

/* 矮螢幕橫向（高度 < 480px，無論寬度）*/
@media (orientation: landscape) and (max-height: 480px) {
  /* 強制改用「左右分欄」棋盤置中佈局，HUD 縮減為極窄條 */
}

/* 觸控裝置（不論尺寸）*/
@media (hover: none) and (pointer: coarse) {
  /* 所有互動熱區強制 min-width/min-height: 44px；取消 hover 效果，改用 active 態回饋 */
}
```

**棋盤自適應縮放邏輯（JS 計算，搭配 CSS）**：

```
可用邊長 = min(
  視窗寬度 - 左右安全邊距*2,
  視窗高度 - 頂部HUD高度 - 底部資訊區高度 - 上下安全邊距*2
)
棋盤容器尺寸 = 可用邊長 (並維持 aspect-ratio: 1/1)
```

此計算於 `window.resize` 與 `orientationchange` 事件時重新觸發，確保旋轉手機或縮放視窗時棋盤即時重新適配，永遠完整可見、永遠不被按鈕/HUD 遮擋。

### 10.4 觸控與點擊優化

- 所有按鈕、棋盤交叉點熱區、滑桿把手，最小尺寸 `44×44px`（符合 Apple HIG 與 Material Design 觸控建議）。
- 按鈕之間最小間距 `8px` 以上，避免手機誤觸相鄰按鈕。
- 棋盤點位若視覺尺寸小於 44px，仍透過**透明擴大熱區**（CSS 偽元素或加大 SVG 點擊範圍）達到最小可觸控標準。

## 11. 音效與背景音樂規格（Web Audio API 合成）

### 11.1 技術方式

全部聲音**不使用任何外部音檔**，改用 `audio-engine.js` 建立一個全域 `AudioContext`，再由 `sfx-synth.js`／`bgm-synth.js` 利用 `OscillatorNode`（音色合成）+ `GainNode`（音量envelope，ADSR）+ `BiquadFilterNode`（音色濾波，模擬鋼琴的圓潤感）即時生成聲音波形並播放。

### 11.2 BGM 規格（鋼琴輕快風格）

- **音色設計**：使用 `triangle` 或 `sine` 波形振盪器疊加少量 `sawtooth` 諧波模擬鋼琴音色的明亮泛音，每個音符套用快速 Attack（0.005s）、適中 Decay、自然 Release 的 ADSR envelope，模擬鋼琴琴鍵敲擊後自然衰減的聽感。
- **曲風**：輕快、明亮的大調（Major Key）旋律，節奏採用中速（約 100–120 BPM），以**琶音（Arpeggio）+ 簡單主旋律**組成的程式化生成旋律迴圈（可寫成一組音高與節拍的資料陣列，由程式依序排程播放，迴圈播放不中斷）。
- **循環設計**：BGM 以 8~16 小節為一個循環單位，循環之間做平滑銜接（最後一個音符與第一個音符的銜接避免突然斷音感）。
- **情境差異**：
  - 主畫面：播放「主題曲」版本（較舒緩、開場感）。
  - 對局中：可切換為「對局版」（節奏略快、更輕快活潑，營造遊戲進行中的愉悅感），或沿用主題曲但音量提升（見 11.5）。
  - 結算畫面（獲勝）：短暫播放歡慶感的鋼琴音效樂句（非循環，播放一次）。
  - 結算畫面（失敗）：短暫播放低沉但不刺耳的鋼琴下行音階（非循環，播放一次，避免使用過度悲傷或恐怖的音效造成負面體驗）。

### 11.3 音效（SFX）規格（高音輕脆）

| 事件 | 音效設計 |
|------|----------|
| 落子（擺子/移動） | 高音域（約 1200–1800Hz）短促 `sine` 音，極短 Decay（約0.08s），清脆「叮」聲 |
| 棋子被選取 | 更短、更輕的高音「滴」聲（約0.04s），音量略低於落子音 |
| 形成 Mill | 三個音符快速上行的高音琶音（如 C6-E6-G6），清亮愉快，明確區別於一般落子音 |
| 吃子（移除對方棋子） | 清脆的高音「噹」聲 + 極短白噪音點綴模擬棋子被拿起的質感 |
| 非法操作提示 | 短促雙音「叮-叮」但音高略低、帶輕微不協和感，提示錯誤但不刺耳 |
| 按鈕點擊（一般 UI） | 極短高音「滴」聲，所有按鈕統一使用，降低音量避免過度疲勞 |
| 切換分頁/開關 toggle | 細微的高音「咔」聲，模擬實體開關觸感 |
| 勝利 | 上行琶音 + 細碎鈴聲疊加（高音域為主） |
| 失敗 | 簡短下行音階，音量與音色仍保持「輕」而非沉重 |
| AI 思考中（loop，極輕量） | 可選：極低音量的高音「滴答」節拍提示思考中（預設關閉，於設定頁可開啟，避免過度干擾） |

所有音效皆以「高音、輕脆、短促」為核心設計原則，避免低頻沉重音效破壞遊戲整體輕快活潑的調性。

### 11.4 音量控制架構

```
最終輸出音量 = 主音量(Master Gain)
             × 類型音量（BGM音量 或 SFX音量，使用者於設定頁調整，預設各75%/85%）
             × 場景倍率（詳見11.5，預設1倍，對局中為5倍）
             × 靜音開關(0或1)
```

以上各層級分別對應獨立的 `GainNode`，串接於音訊路由鏈中（`Oscillator → 場景倍率Gain → 類型音量Gain → 主音量Gain → AudioContext.destination`），方便各層獨立調整且不互相干擾。

### 11.5 對局中 BGM 音量 ×5 規格（對應需求 7）

- 定義一個「場景倍率」變數 `bgmSceneMultiplier`：
  - 於主畫面、說明頁、設定頁：`bgmSceneMultiplier = 1`（一般音量，避免在非對局畫面就過於吵雜）。
  - **進入對局畫面（`view-game` 啟用）的瞬間，`bgmSceneMultiplier` 立即調整為 `5`**，並透過 `GainNode.gain` 的 `linearRampToValueAtTime` 在約 0.5 秒內平滑過渡到 5 倍音量，避免瞬間音量暴增造成刺耳的突兀感。
  - 離開對局畫面（返回主選單）時，平滑過渡回 `1` 倍。
- **重要實作備註**：由於原始 BGM 合成音量已先以「合理可聽」的基準音量設計（例如基準峰值約 -24dB 至 -18dB 範圍），５倍增益是相對於此基準的**倍數放大**，而非「絕對音量無限增大」；程式中仍須加入**最終輸出限制（Limiter / 動態壓縮 `DynamicsCompressorNode`）**，確保即使乘以 5 倍增益，最終輸出仍不會超過 0dB 造成破音（clipping）或對使用者聽力/裝置喇叭造成損害。此為安全性與音質的必要防護機制，需在規格書中明確要求實作。
- 設定頁的「背景音樂音量」滑桿仍對使用者可見且可調整的是**使用者基準音量（0–100%）**，「對局中5倍」是疊加在此基準之上的場景效果，兩者疊加邏輯需在程式註解中清楚標示，方便日後維護調整倍率數值。

### 11.6 iOS / 行動裝置音訊啟動相容性

- 因瀏覽器政策限制（尤其 iOS Safari），`AudioContext` 必須在「使用者手勢（如 click/touchstart）」中才能 `resume()`。實作上需在**使用者第一次點擊任何按鈕**（不論是主畫面的開始遊戲、設定、說明）時，立即觸發一次 `audioContext.resume()`，確保後續所有音效與 BGM 都能正常播放，不會出現「手機上完全沒聲音」的問題。

## 12. AI 演算法規格（Minimax + Alpha-Beta，三難度）

### 12.1 整體架構

所有難度共用 `ai-evaluator.js` 提供的**盤面評估函式（Heuristic Evaluation）**，差異在於：(a) 搜尋深度、(b) 是否使用 Alpha-Beta 剪枝、(c) 是否加入隨機性、(d) 評估函式權重的細緻程度。

### 12.2 盤面評估函式（Heuristic）設計

評估分數 = 以下各項加權總和（正值對 AI 有利，負值對玩家有利）：

| 評估項目 | 說明 | 權重級別 |
|----------|------|----------|
| 棋子數量差 | (AI棋子數 − 玩家棋子數) | 高權重 |
| 可形成 Mill 的潛力位（Open Mill / 兩子一線） | 己方已有兩子同線且第三點為空 | 中高權重 |
| 對方可形成 Mill 的潛力位 | 需優先防守的威脅點 | 中高權重（負向扣分） |
| 雙重威脅位（Double Mill Threat） | 一個落點同時能完成兩條 Mill | 高權重 |
| 棋子機動性（可移動的合法走法數量） | 走法越多越靈活，尤其在移動階段重要 | 中權重 |
| 關鍵位置（交叉點連接數）佔據 | 棋盤上「十字交叉點」（連接4條邊的點，如中圈四邊中點）價值高於「角點」（僅連接2條邊） | 中權重 |
| 對方被卡死風險 | 若可預判對方下一步無合法走法，給予極高分 | 極高權重（接近勝利判斷） |
| 飛行階段特殊評估 | 雙方剩3子時，著重「能否形成可重複開合的 Mill」 | 特殊規則 |

### 12.3 簡單難度（Easy）— `ai-easy.js`

- **不使用** Minimax 樹搜尋，採用「規則式啟發法（Rule-based Heuristic）+ 隨機性」：
  1. 若有「一步即可形成 Mill」的走法 → 70% 機率採用（刻意保留 30% 機率不採用，模擬新手會漏看機會）。
  2. 若沒有，檢查「玩家下一步是否能形成 Mill」→ 若有威脅，50% 機率進行防守（其餘 50% 機率忽略威脅，模擬新手疏忽)。
  3. 若以上皆無，從所有合法走法中**隨機選擇**（但排除明顯送子的選項，如送到讓玩家立即吃掉且無任何補償的走法，避免 AI 顯得「太蠢」而破壞遊戲性）。
- 反應時間：刻意加入 0.3~0.6 秒的「思考延遲」（非真的需要運算，是 UX 設計，讓 AI 顯得有在思考，避免瞬間落子顯得不自然）。

### 12.4 普通難度（Normal）— `ai-normal.js`

- 採用 **Minimax 演算法**（不一定需要剪枝，視效能而定，建議仍加入 Alpha-Beta 以保持流暢），**搜尋深度 2~3 層**。
- 評估函式採用 12.2 表格中**部分權重**（棋子數量差、Open Mill潛力、基本防守），不考慮「雙重威脅」「卡死風險」等進階評估，使其在中後段對局會出現可被人類抓到漏洞的合理失誤，符合「普通」難度的定位。
- 反應時間：思考延遲 0.5~1 秒。

### 12.5 困難難度（Hard）— `ai-hard.js`

- 採用完整 **Minimax + Alpha-Beta 剪枝**，搜尋深度依遊戲階段動態調整：
  - 擺子階段：深度 5（因分支因子較大，避免運算過久）。
  - 移動階段（棋子數量較少、分支因子降低）：深度 7。
  - 飛行階段（剩 3 子，分支因子小）：深度可加大到 9~11（甚至可考慮在此階段做近似完全搜尋）。
- 評估函式採用 12.2 表格**全部權重**，並針對「雙重威脅」「對方卡死風險」給予顯著加權，使 AI 具備主動製造雙重威脅、預判玩家陷入死局等進階戰術能力。
- **走法排序優化（Move Ordering）**：搜尋前先將「可形成Mill的走法」「可阻擋對方Mill的走法」排在搜尋順序前面，提升 Alpha-Beta 剪枝效率，加速運算。
- **效能保護機制**：由於不使用 Web Worker（考量免 build/server 環境的相依性風險，見 2.2），困難難度的深層搜尋需採用**時間切片（Time-slicing）**方式執行：將 Minimax 遞迴搜尋拆分為多個 `setTimeout(fn, 0)` 區塊逐步執行，避免長時間佔用主執行緒造成畫面凍結（UI 卡死）；並設定**最長運算時間上限（如 3 秒）**，超過時間則回傳目前搜尋到的最佳結果（Iterative Deepening 概念：由淺到深逐層搜尋，每完成一層都記錄當前最佳解，時間到即回傳當前最佳解，而非整段拋棄重來）。
- 反應時間：實際運算時間依盤面複雜度可能落在 0.5~3 秒，UI 上以「思考中動畫」呈現，避免使用者誤以為當機。

### 12.6 難度切換時機

- 難度於「開始遊戲」彈窗中選擇，亦可於設定頁設定「預設難度」（下次開始遊戲時預先選中該選項，仍可在彈窗中臨時更改）。
- 對局進行中**不可變更難度**（避免邏輯混亂），如需變更須透過「重新開始」或返回主選單重新開局。

## 13. 本地儲存與「繼續遊戲」規格

### 13.1 儲存內容與 Key 命名

使用 `localStorage`，所有 key 統一加上專案前綴 `nmm_` 避免與其他網站/應用衝突：

| `nmm_first_launch` | 是否為首次啟動（決定是否自動彈出簡短新手導覽提示） | `"true"` / `"false"` |

### 13.2 `nmm_save_game` 資料結構


- `version`：存檔格式版本號，未來規則或結構若有調整，可用此欄位做向後相容檢查（若版本不符，提示使用者「存檔格式已更新，無法讀取舊存檔」並清除舊存檔，避免讀取錯誤資料造成程式崩潰）。
- `history`：記錄完整走步歷史，主要用途為未來可能擴充的「步驟回顧」功能，非必要欄位但建議保留以利除錯與未來功能擴充。

### 13.3 「繼續遊戲」運作邏輯

1. 主畫面載入時，檢查 `nmm_save_game` 是否存在且 `gameOver: false` → 決定「繼續遊戲」按鈕是否可點擊。
2. 點擊「繼續遊戲」→ 讀取 JSON → 還原 `GameState` 物件 → 切換至 `view-game` → 依存檔還原棋盤渲染、回合提示、雙方剩餘棋子。
3. **自動存檔時機**：每次完成一個完整動作（落子完成/移動完成/吃子完成，回合即將切換之前）即自動寫入 `localStorage`，確保即使使用者直接關閉瀏覽器分頁，也能在下次開啟時從最後一個完整動作點繼續，不會有「存到一半棋子消失」的問題。
4. 遊戲結束（任一方獲勝）時，將 `gameOver` 設為 `true` 並寫入結果，主畫面「繼續遊戲」按鈕應呈現禁用狀態（因為已無「未完成」對局）。
5. 開始新遊戲（從主畫面「開始遊戲」進入）會覆蓋 `nmm_save_game`，需要先做二次確認（見 6.3）。

### 13.4 容錯處理

- 讀取存檔時以 `try/catch` 包裹 `JSON.parse()`，若解析失敗（資料毀損）則視為無存檔，並清除該毀損的 key，避免每次啟動都載入失敗。
- `localStorage` 在部分瀏覽器的隱私模式下可能無法寫入（拋出例外），需捕捉例外並降級為「僅記憶體內暫存（重新整理頁面即消失）」，同時於設定頁顯示提示文字告知使用者「目前瀏覽器無法儲存進度，建議使用一般模式開啟」。

## 14. 多國語系規格（中 / 英 / 日）

### 14.1 架構設計

語言包不透過 `fetch` 載入（理由見 2.2），而是直接以獨立 JS 檔案宣告為全域物件常數：

```javascript
// js/i18n/lang-zh.js
window.LANG_ZH = {
  app_title: "九子棋",
  menu_start: "開始遊戲",
  menu_continue: "繼續遊戲",
  menu_howto: "遊戲說明",
  menu_settings: "設定",
  difficulty_easy: "簡單",
  difficulty_normal: "普通",
  difficulty_hard: "困難",
  game_status_player_place: "請點擊空格放置棋子",
  game_status_player_move: "請選擇要移動的棋子",
  game_status_mill_formed: "你形成了三連線！請選擇要吃掉的對方棋子",
  game_status_ai_thinking: "電腦思考中…",
  // ……（依全站所有顯示文字逐一列出對應 key）
};
```

`lang-en.js`、`lang-ja.js` 採用**完全相同的 key 結構**，只替換對應翻譯內容，確保 `i18n-manager.js` 可用單一套邏輯切換，不需要針對語言寫條件判斷。

### 14.2 `i18n-manager.js` 運作邏輯

1. 啟動時讀取 `nmm_settings.language`（若無設定，預設依瀏覽器 `navigator.language` 自動判斷：偵測到 `zh` 開頭 → 中文；`ja` 開頭 → 日文；其餘 → 英文，找不到對應則 fallback 英文）。
2. 提供全域函式 `t(key)`：回傳目前語言對應的字串；若該 key 在目前語言包中不存在，fallback 至英文語言包，仍找不到則回傳 `key` 本身（方便開發階段除錯，不會讓畫面整片空白）。
3. 所有需要顯示文字的 DOM 元素統一加上 `data-i18n="key名稱"` 屬性，`i18n-manager.js` 於切換語言時掃描所有帶有此屬性的元素並批次更新其 `textContent`（或 `placeholder`、`aria-label` 等視元素類型而定，需支援 `data-i18n-attr` 指定要更新的屬性）。
4. 切換語言為**即時生效**，不需要重新整理頁面；切換後立即寫入 `nmm_settings.language` 做持久化。

### 14.3 翻譯內容覆蓋範圍

語言包需完整覆蓋以下所有區域的文字（不可遺漏，避免切換語言後出現「混雜語言」的破碎體感）：

- 主畫面所有按鈕與標題
- 遊戲畫面所有 HUD 文字、狀態提示句、彈窗文字（確認框、結算畫面勝/負文案）
- 說明頁全部 7 個分頁的標題、條列重點、Q&A 問答內容
- 設定頁所有分組標題、選項名稱、二次確認彈窗文字
- 難度名稱、主題名稱（如「經典木紋」「海洋」「夕陽」「森林」「夜間」亦需各語言翻譯）
- 各類錯誤提示、Toast 輕提示文字

### 14.4 排版隨語言調整的注意事項

- 日文、中文字符較寬，英文單字長度差異大（例如「設定」vs「Settings」），按鈕與卡片需使用**彈性寬度（非固定 px 寬度）**設計，內容置中且有足夠左右 padding，避免英文版按鈕文字溢出或日文版按鈕過度留白不對稱。
- 字體堆疊（見 10.1 `--font-family-base`）已包含中日文皆能正確顯示的字型，避免切換語言後出現「缺字變方框」的問題。

## 15. 動畫與互動細節（對應需求 13：擬真、豐富）

### 15.1 棋子質感

- 棋子使用多層 `radial-gradient` 疊加，模擬「上方有光源照射的球體/圓盤」光影效果（高光點偏左上、陰影偏右下），並加上極細的外框陰影（`box-shadow`）模擬棋子放置在棋盤上的立體浮起感。
- 棋子材質可依主題微調：例如「經典木紋」主題的棋子可疊加細微木紋紋理（CSS `background-blend-mode` 疊加雜訊紋理圖樣，或用 SVG `<filter>` 產生雜訊濾鏡），「夜間」主題棋子可改為類似玉石/水晶的半透光質感。

### 15.2 棋盤質感

- 棋盤底色依主題可呈現「木紋桌面」「石材」「布質棋盤」等質感聯想（透過漸層 + 細微紋理雜訊模擬，非真實照片貼圖，保持輕量與向量銳利度）。
- 棋盤四周可加上極簡的「外框裝飾」（如木框立體感的 `box-shadow` 內外陰影組合），讓棋盤看起來像「擺在桌上的實體棋盤」而非單純扁平線稿。

### 15.3 互動回饋動畫

| 互動 | 動畫效果 |
|------|----------|
| 滑鼠/手指懸停於可落子點 | 該點位浮現淡色光暈圈，提示「這裡可以下」 |
| 點擊選取己方棋子 | 棋子微微放大 + 呼吸光暈外框 |
| 棋子移動 | 沿棋盤連線路徑平滑滑動（非瞬間跳轉），時間約 0.3~0.4 秒，使用 `ease-in-out` |
| 形成 Mill | 連線描邊動畫（線條像被點亮一樣由兩端棋子向中間匯聚發光），同時三顆棋子做一次「彈跳強調」 |
| 吃子瞬間 | 被吃掉的棋子先短暫放大閃爍紅光，再縮小淡出消失（並非直接瞬間消失，保留視覺確認感） |
| 遊戲結束 | 勝方棋子整排（或全部存活棋子）做一輪「慶祝跳動」波浪動畫，敗方棋子轉為灰階 |
| 畫面切換（View 之間） | 淡入淡出 + 輕微位移（slide），時間約 0.25 秒，避免過長動畫造成操作延遲感 |

### 15.4 主畫面背景動態效果

- 背景採用緩慢飄浮的棋子裝飾圖形（純裝飾、無互動性），透過 CSS `@keyframes` 做極緩慢（15~30秒一個循環）的位移與透明度呼吸變化，營造「畫面豐富有趣」但不會搶走主按鈕的視覺焦點，且效能消耗極低（避免行動裝置耗電/發熱問題）。

## 16. 無障礙與防呆設計

- 所有可互動元素（按鈕、棋盤點位、滑桿）皆加上適當的 `aria-label`（依目前語言動態更新），確保螢幕報讀軟體可正確朗讀。
- 色彩不是唯一辨識依據：玩家棋子與 AI 棋子除顏色不同外，額外加上**形狀內標記**（如圓點 vs 三角形小圖示）區分，照顧色盲使用者。
- 所有「不可逆」操作（清除存檔、投降、開新局覆蓋舊局）皆需二次確認彈窗，且確認按鈕與取消按鈕視覺上要有明確差異（危險操作用紅色強調 + 取消按鈕為預設聚焦狀態，避免使用者手滑誤觸危險選項）。
- 鍵盤可操作性：主要彈窗與按鈕支援 `Tab` 鍵切換焦點、`Enter`/`Space` 觸發，棋盤亦可選配方向鍵移動焦點 + Enter 落子（進階加分項，非必需但建議納入）。
- AI 思考期間明確鎖定玩家操作（所有棋盤熱區暫時 `pointer-events: none` 並降低透明度提示），避免使用者誤以為卡住而重複亂點。
