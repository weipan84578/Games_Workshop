## 1. 專案總覽

### 1.1 遊戲簡介
「Basketball Shoot 投籃機」是一款模擬街機投籃機台的網頁小遊戲。玩家於限定時間內，透過拖曳／點擊方式調整力道與角度，將籃球投入籃框得分，並挑戰個人與全球最高分排行榜。

### 1.2 核心設計原則
| 原則 | 說明 |
|---|---|
| 零安裝、零建置 | 使用者只需雙擊 `index.html` 即可遊玩，不需 npm install、不需啟動任何 local server |
| 行動裝置優先 | 觸控操作與版面配置需在手機直向/橫向皆順暢，操作按鍵不可遮擋畫面 |
| 高可讀性 UI | 全站大字體、清晰配色、可愛風插畫 |
| 模組化結構 | CSS / JS 依功能分類至獨立資料夾，`index.html` 僅負責引入 |
| 多語系 | 中 / 英 / 日 即時切換，無需重新整理頁面 |
| 可測試 | 每個核心邏輯模組皆須有對應單元測試 |

### 1.3 目標裝置
- 桌機瀏覽器（Chrome / Edge / Firefox / Safari 最新版）
- 手機／平板（iOS Safari、Android Chrome），支援直向與橫向

## 2. 技術架構規範

### 2.1 技術限制（重要）
- ❌ 禁止使用任何需要 build 步驟的框架（React/Vue/Webpack/Vite 等一律不可）
- ❌ 禁止使用 Node.js server、禁止使用 `npm start`
- ✅ 僅使用 **原生 HTML5 + CSS3 + JavaScript (ES6+)**
- ✅ 圖形渲染使用 `<canvas>` 搭配 2D Context（確保效能與流暢度）
- ✅ 音效播放使用原生 `Web Audio API` 或 `<audio>` 標籤
- ✅ 資料儲存（設定、進度、排行榜）使用 `localStorage`
- ✅ 所有第三方函式庫（如需要）必須是可離線運作的本地檔案，不可依賴 CDN 連線（避免離線無法遊玩）

### 2.2 相容性
- 支援 ES6 modules（`<script type="module">`），開啟 `index.html` 時以 `file://` 協定亦可正常運作
  - 若瀏覽器對 `file://` 下 ES module 有 CORS 限制，需改用傳統 `<script src="...">` 逐一載入並以全域命名空間（如 `window.BB.xxx`）避免衝突，確保「直接雙擊開啟」的需求 100% 成立
- 不依賴任何後端 API

## 4. RWD 響應式設計規範

### 4.1 斷點定義
| 裝置類型 | 寬度範圍 | 版面策略 |
|---|---|---|
| 手機直向 | ≤ 480px | 單欄堆疊、操作按鍵固定於畫面下方安全區外 |
| 手機橫向 | 481px–767px | 遊戲畫面置中，操作鍵移至左右兩側邊緣 |
| 平板 | 768px–1024px | 保留桌面版比例，按鈕加大觸控熱區 |
| 桌機 | ≥ 1025px | 完整版面，可顯示側邊資訊欄（連擊數、歷史最高分） |

### 4.2 核心原則：操作介面不得遮擋遊戲畫面
- 遊戲主畫布（canvas）永遠保持在「安全可視區域」（safe-area）內置中顯示，比例以 `aspect-ratio` 鎖定，避免變形
- 行動裝置的虛擬操作鍵（拖曳投籃區、暫停鍵）一律採用：
  - **半透明浮動圖層**（opacity 建議 0.55–0.75），置於畫布下緣 12% 高度的「操作帶」內，不疊加於籃框、球體有效範圍之上
  - 使用 `env(safe-area-inset-bottom)` 避開 iPhone 瀏海/手勢列
  - 支援旋轉自動偵測（`orientationchange`），橫向時操作鍵自動移至左右兩側，避免蓋住中央畫面
- HUD（分數、倒數計時）固定於畫面**最上方**細長條，不與操作區重疊

### 4.3 觸控與滑鼠共用邏輯
- 統一使用 Pointer Events（`pointerdown/move/up`），同一套邏輯同時支援滑鼠拖曳與手指觸控
- 拖曳投籃：按下起始點 → 拖曳方向與距離換算為角度與力道 → 放開瞬間發射
- 禁止使用僅支援桌機的 `mouseover` 作為必要操作觸發（避免行動裝置無法遊玩）

### 4.4 版面自適應細節
- 所有容器使用 `clamp()`、`vw`/`vh`、`%` 為主，避免寫死 px 造成小螢幕溢出
- 彈窗（說明/設定）在手機上改為全螢幕滑出式，桌機則為置中彈窗，皆可捲動且不裁切內容

## 5. 視覺設計規範（字體 / 配色 / 主題）

### 5.1 字體規範
- 全站字體大小 **一律提高**，基準：
  - 內文：`18px`（手機）／`20px`（桌機）起跳
  - 按鈕文字：`22px` 以上，粗體
  - 標題：`32px`–`48px`，粗體，加陰影或描邊增加可讀性
  - 分數／計時器：`40px` 以上，等寬數字字體，確保跳動時不晃動版面
- 字體需清晰明確，優先使用高辨識度圓體/黑體（例如：思源黑體 / Noto Sans TC / Poppins / Baloo 2 可愛圓體），並將字型檔內嵌於 `assets/fonts`，確保離線可用
- 所有文字皆需有足夠對比度（WCAG AA 以上，對比度 ≥ 4.5:1），避免「看不見文字」情況

### 5.2 配色系統（多主題可選）
提供 **5 套配色主題**，於設定頁可即時切換並即時預覽：

| 主題 | 說明 | 主色調 |
|---|---|---|
| 🎨 預設經典 | 籃球場橘＋木地板棕 | 橘 #FF7A00 / 深棕 #4A2E1D |
| 🌸 可愛風（重點主題） | 大量可愛圖案：愛心籃球、笑臉雲朵、貼紙風星星、粉彩漸層 | 粉紅 #FFB6D9 / 薄荷 #B5F2E0 / 奶油黃 #FFF3B0 |
| 🌙 深色模式 | 護眼深色，霓虹強調色 | 深藍黑 #1B1F3B / 螢光藍 #4CC9F0 |
| 🌊 海洋藍 | 清爽藍白 | 海藍 #0077B6 / 白 #F1FAFB |
| 🌇 夕陽橘 | 溫暖漸層 | 珊瑚橘 #FF6F59 / 紫 #7B4B94 |

**可愛風主題細節**（因需求特別強調）：
- 主畫面背景加入飄浮愛心、星星、笑臉雲朵等 SVG 裝飾動畫（緩慢漂浮，不干擾操作）
- 按鈕造型改為圓角糖果形，hover/點擊有 Q 彈縮放動畫
- 籃球、籃框加上可愛表情貼圖選項（可於設定切換「可愛球皮膚」）
- 得分時彈出可愛角色歡呼插畫 + 貼紙風文字特效（如「Nice Shot!💖」）

### 5.3 配色安全規則
- 所有主題皆需檢查文字與背景對比，避免「淺色字疊淺色底」或「按鈕與背景同色系無邊界」問題
- 強調色（CTA 按鈕）在每個主題中都需與背景形成明顯區隔（邊框 + 陰影 + 對比色）

## 6. 主畫面規格

### 6.1 主畫面原則（簡潔乾淨）
- 主畫面**僅顯示 4 個核心功能按鈕**，不放任何設定選項（配色、音量調整等一律收在「設定」頁面內）：
  1. ▶️ **開始遊戲**：清除進度，開新局
  2. ⏯️ **繼續遊戲**：讀取 `localStorage` 中斷點存檔（無存檔時此按鈕顯示為灰階不可點擊，並附提示文字）
  3. 📖 **說明**：開啟遊戲玩法彈窗
  4. ⚙️ **設定**：開啟設定彈窗
- 背景可播放可愛風/預設風動態插畫（依使用者當前選擇的主題呈現），Logo 置中偏上，四按鈕垂直置中排列，行動裝置維持大按鈕（觸控熱區 ≥ 48x48px，建議 64x64px）
- 主畫面右上角僅保留「語言切換」小型下拉（因多語系需求需全站可存取，不算作設定選項，屬於基本可及性功能）

### 6.2 按鈕狀態
| 按鈕 | 一般狀態 | 無存檔時 | 點擊回饋 |
|---|---|---|---|
| 開始遊戲 | 高亮主色 | - | 縮放 + 音效 |
| 繼續遊戲 | 高亮次色 | 灰階 + disabled | 縮放 + 音效（若可點擊）|
| 說明 | 中性色 | - | 縮放 + 音效 |
| 設定 | 中性色 | - | 縮放 + 音效 |

## 7. 遊戲畫面規格

### 7.1 畫面配置
```
┌─────────────────────────────────────┐
│  HUD：分數 | 剩餘時間 | 連擊數 | 暫停鍵  │ ← 頂部固定，不遮擋球場
├─────────────────────────────────────┤
│                                       │
│         籃框 / 籃網 / 計分區            │
│                                       │
│         拋物線輔助線（可選開關）          │
│                                       │
│         籃球 / 發射起始點                │
│                                       │
├─────────────────────────────────────┤
│   （行動裝置限定）半透明拖曳操作帶          │ ← 不遮擋球場主要範圍
└─────────────────────────────────────┘
```

### 7.2 玩法邏輯
- 玩家於畫面下方拖曳籃球，方向與拉力決定發射角度與初速
- 物理引擎（`physics.js`）計算拋物線軌跡，含重力加速度、空氣阻力簡化模擬
- 進籃判定：球心進入籃框有效區域且下墜方向正確 → 得分
  - 空心球（未觸碰籃框邊緣）：+3 分 + 連擊加成
  - 擦板進球：+2 分
  - 連續進球：連擊倍率（x1.2 / x1.5 / x2）
- 時間倒數結束或生命值歸零 → 結算畫面，寫入排行榜

### 7.3 擬真細節（需求 13）
- 籃球具備真實旋轉動畫（依飛行方向旋轉貼圖）
- 籃網進球時有 Q 彈晃動動畫（可用簡易骨骼點模擬或彈簧動畫）
- 球彈框時依真實角度反彈（簡化碰撞反射公式）
- 加入陰影投射（球的地面陰影隨高度縮放，增加空間深度感）
- 背景加入輕微視差捲動（觀眾看台/燈光閃爍等細節裝飾，非必要互動元素）

### 7.4 暫停與繼續
- 點擊暫停鍵 → 遊戲狀態機切換為 `PAUSED`，彈出半透明遮罩 + 繼續/回主選單/重新開始選項
- 離開頁面前自動存檔目前分數/時間/關卡進度至 `localStorage`，供主畫面「繼續遊戲」使用

## 8. 音效與 BGM 規格

### 8.1 BGM 規範
- 提供 **至少 4 首鋼琴輕快風格 BGM**，隨機或依玩家於設定中選擇的清單循環播放：
  1. `bgm_piano_morning.mp3`（輕快、明亮）
  2. `bgm_piano_bounce.mp3`（節奏感、俏皮）
  3. `bgm_piano_cute.mp3`（可愛風主題專用，配合可愛主題自動套用）
  4. `bgm_piano_victory.mp3`（結算/破紀錄時播放）
- BGM 需支援無縫循環（loop seamless，避免銜接處雜音）

### 8.2 音效規範
- 使用**高音、輕脆**類型音效（如：鈴鐺聲、清脆鋼琴單音、水晶碰撞聲），涵蓋：
  - 按鈕點擊音
  - 籃球出手音
  - 空心進球音（清亮鈴聲）
  - 擦板進球音
  - 連擊音（音高隨連擊數遞增，營造爽感）
  - 倒數計時音（最後 3 秒提示音）
  - 遊戲結束/破紀錄音

### 8.3 音量規範（重點需求）
> ⚠️ 需求第 7 點：**「遊戲中的 BGM 音量都放大為原來的 10 倍」**

- 實作方式：`audio-manager.js` 內定義 `BASE_BGM_VOLUME` 與 `BGM_VOLUME_MULTIPLIER = 10`
- 由於 Web Audio 音量正常範圍為 `0.0 ~ 1.0`，「放大 10 倍」實作規則如下：
  - 使用 `GainNode` 控制音量，最終增益值 = `baseVolume * 10`，並以 `Math.min(gainValue, GAIN_HARD_CAP)` 做安全上限保護（`GAIN_HARD_CAP` 建議設為 `3.0`，避免破音/爆音傷害使用者聽力與裝置喇叭）
  - 設定頁的 BGM 音量滑桿仍以「使用者感知音量 0–100%」呈現，內部運算時套用 10 倍增益公式，並顯示提示文字：「遊戲內 BGM 音量已依設計加成，此滑桿為額外微調」
  - 需於單元測試中驗證：`getFinalGain(userVolume)` 回傳值確實為 `userVolume * 10`（經上限裁切前）
  - **安全性附註**：因應聽力保護，實際輸出音量仍會通過 `GAIN_HARD_CAP` 限制與音訊壓縮（compressor node）處理，避免對使用者造成聽力不適或喇叭損壞風險；此上限與壓縮邏輯視為本規格之必要安全設計，不可移除

### 8.4 音效系統技術要求
- 使用 Web Audio API 的 `AudioContext` + `GainNode` 分別控制 BGM 軌與 SFX 軌（雙獨立音量通道）
- 需處理瀏覽器「使用者互動前禁止自動播放」限制：首次點擊任一按鈕時才 `resume()` AudioContext

## 13. 動畫與效能規格

### 13.1 流暢度標準
- 遊戲主迴圈使用 `requestAnimationFrame`，目標 **60 FPS**
- 避免每幀進行大量 DOM 操作；球體/籃網等動態元素優先使用 Canvas 繪製而非大量 DOM 節點
- 圖片資源需壓縮（PNG 壓縮 / SVG 優先），避免載入延遲造成卡頓
- 音效／BGM 需預先載入（preload），避免播放當下產生延遲

### 13.2 動畫規範
- 按鈕互動：`transform: scale()` + `transition` 控制在 100–200ms 內完成，避免過度延遲感
- 進球特效、連擊特效等使用 CSS `@keyframes` 或 Canvas 粒子效果，避免使用會造成 reflow 的屬性（如直接改 `width/height`，改用 `transform`）
- 行動裝置需測試低階裝置效能，必要時提供「特效品質」設定（高/中/低）於設定頁，降低粒子數量以維持流暢度
